# html2md 设计文档

**更新日期：** 2026-04-15

## 背景与目标

当前项目已经从“单文件 HTML 转 Markdown 页面”演进为一个面向单用户、本机浏览器使用的书籍管理器。它的核心目标不再是一次性转换单个 HTML，而是帮助用户把多章 HTML 内容沉淀为一本结构化书籍，并在保存时持续生成 Markdown，最终打包导出。

**目标：**
- 提供一个本地书架，用于创建、进入、删除书籍
- 提供单本书工作台，用于管理章节、编辑 HTML、预览 Markdown
- 在章节保存时自动调用转换器生成派生 Markdown
- 提供链接采集能力，支持正文提取与页面片段选取
- 支持整本书导出为 zip，内含章节 Markdown、`README.md`、`toc.json` 与完整备份 `book.json`
- 支持通过单个 `book.json` 将整本书重新导入书架并继续编辑
- 在视觉上保持“阅读产品”而非“开发工具”的体验，采用蓝白主色、扁平化、内容优先的设计

**非目标：**
- 不支持登录、多用户、云同步或协作编辑
- 不支持 Markdown 反向编辑 HTML
- 不支持卷/章多级嵌套，章节结构保持单层有序列表
- 不引入服务端数据库或文件持久化

## 技术栈

- **框架：** Next.js App Router
- **语言：** TypeScript
- **运行时：** React 19 + Next.js 15
- **样式：**
  - Tailwind CSS 4
  - 自定义 CSS 变量主题
  - Ant Design 用于模态框与消息提示
- **核心库：**
  - `turndown`
  - `turndown-plugin-gfm`
  - `jszip`

## 信息架构

### `/`

首页为书架页，负责：
- 展示全部书籍
- 新建书籍
- 导入 `book.json`
- 删除书籍
- 进入单本书工作台

### `/?bookId=<id>`

首页同时承担书架页与书籍工作台的路由入口：
- 未传 `bookId` 时展示书架
- 传入 `bookId` 时展示对应书籍工作台

这样可以避免动态路由依赖，兼容 GitHub Pages 的纯静态部署。

## 持久化设计

### IndexedDB

项目使用 IndexedDB 作为唯一持久化层，不引入后端存储。

**对象存储：**
- `books`
- `chapters`

**仓储职责：**
- 查询书籍摘要
- 创建、删除、重命名书籍
- 导入整本书备份
- 创建、删除、重命名章节
- 调整章节顺序
- 更新章节 HTML，并同步生成 Markdown

### 设计原因

- 满足单用户、本地使用场景
- 降低部署复杂度
- 保持导出功能纯前端可用

## 核心交互流程

### 1. 创建书籍

用户在书架页点击“新建书籍”，通过模态框输入书名。确认后：
- 生成书籍记录
- 更新书架列表
- 自动跳转到 `/?bookId=<id>` 对应的工作台

### 2. 创建与管理章节

工作台左侧为章节管理区，支持：
- 新建章节
- 重命名章节
- 删除章节
- 上移、下移章节顺序

章节采用单层顺序列表，不支持嵌套。

### 3. 编辑与自动保存

当前章节在右侧工作区编辑：
- 编辑态：HTML 输入
- 预览态：Markdown 预览

每次输入变化后：
- 若内容与持久化内容不同，则进入“未保存”状态
- 经过 800ms 防抖后自动保存
- 保存时调用 `convert(html)` 生成最新 Markdown
- 保存成功后更新书籍 `updatedAt`

切换章节前会先尝试冲刷待保存草稿，避免丢失。

### 4. 导出整本书

导出时：
- 先冲刷当前草稿
- 重新读取该书与全部章节
- 生成 zip 文件

zip 内内容：
- 每章一个 `.md`
- `README.md`
- `toc.json`
- `book.json`

其中 `book.json` 保存整本书的完整编辑态数据，用于后续重新导入：
- 书籍标题与时间戳
- 各章节顺序
- 各章节原始 HTML
- 各章节派生 Markdown

### 5. 导入整本书

导入时：
- 用户在书架页选择单个 `book.json`
- 系统校验其结构与版本
- 为导入结果生成新的书籍与章节 id
- 将其写入 IndexedDB
- 导入成功后自动进入 `/?bookId=<id>` 对应的工作台

章节文件命名规则：
- `01-章节名.md`
- `02-章节名.md`

文件名会做安全清洗，并对重复标题追加序号。

### 6. 链接采集

链接采集以弹窗方式接入工作台，而不是常驻在主编辑区中。

当前支持两条路径：

- 正文提取
  - 直接在服务端抓取目标 HTML
  - 抽取文章主体
  - 进入预览，再确认导入
- 页面片段选取
  - 通过服务端代理重写页面
  - 在同源 `iframe` 中进行 DOM 选取
  - 进入预览，再确认导入

两条路径统一进入预览态，只有点击“导入正文”后，才会覆盖当前章节 HTML。

### 7. 页面片段选取的层级上溯

片段选取不只保存当前点击节点，还会同时构造其祖先链，直到 `body` 为止。

预览结构会保存：

- 当前层级 HTML
- 当前层级完整标签路径
- 从当前节点到父节点的全部候选层级
- 当前停留层级索引

预览区据此支持：

- 选择父级
- 选择子级
- 导入当前层级

### 8. 站点风控识别

对公开网页的代理采集并不等于一定能拿到正文。

服务端会优先识别以下风控/验证特征：

- `x-waf-uuid`
- WAF 相关 cookie
- 极小空壳 HTML
- `probe.js`
- captcha / verification 关键词

如果命中，接口直接返回明确错误，而不是继续走 iframe 逻辑后再报超时。

### 9. 正文提取设计

正文提取不是完整的通用阅读器，而是轻量启发式抽取。

当前策略：

- 优先匹配 `article` / `main` / 常见正文类名
- 对候选节点按文本长度、段落数、标题数、图片数、列表数评分
- 移除导航、评论、推荐、广告等噪音区块
- 恢复图片、链接等资源的绝对地址

负责领域层纯逻辑：
- 书名默认值规范化
- 章节标题默认值规范化

## 视觉设计

### 设计方向

整体视觉参考阅读类产品，而不是后台管理系统：
- 蓝白主色
- 扁平化
- 低装饰
- 内容优先
- 卡片与面板使用纯色底 + 细边框 + 少量阴影
- Markdown 预览区域采用更接近书页阅读的排版节奏，保留列表符号、编号与层级

### 色彩策略

**浅色模式：**
- 页面背景：极浅蓝灰
- 面板背景：白色
- 强调色：中蓝
- 文本：深蓝灰

**深色模式：**
- 页面背景：深蓝黑
- 面板背景：深蓝色块
- 强调色：浅蓝
- 文本：高对比浅色

### 结构原则

- 首页头部高度控制在较紧凑范围，不做大 Hero Landing
- 工作台为固定视口布局，避免整页滚动造成阅读区不稳定
- Markdown 预览滚动收敛到内容区内部
- 章节选中状态必须一眼可辨

### 深色模式策略

项目使用 `.dark` class 驱动深色模式，而不是仅依赖系统媒体查询。

**原因：**
- 用户可显式切换
- 组件层可保持主题一致
- Ant Design 主题可跟随同步切换

## 设计决策

### 1. 保留转换核心，替换产品外壳

转换器依然是产品核心能力，但从“单文件工具页”升级为“章节持久化工作流”的底层引擎。

### 2. 选择 IndexedDB 而不是服务端

当前需求明确是本地单用户工具，IndexedDB 复杂度最低，且足够支持书籍与章节管理。

### 3. Markdown 只作为派生数据

避免 HTML / Markdown 双向编辑导致状态冲突。HTML 是源数据，Markdown 是自动生成结果。

### 4. 导出 zip 采用轻量依赖

使用 `jszip` 让导出流程可靠、跨浏览器、可维护。

### 5. 视觉采用“扁平化蓝白阅读工具”

### 6. 路由改为查询参数以兼容 GitHub Pages

GitHub Pages 只能托管静态资源，不适合依赖服务端 API Route 或运行时动态路径参数。

因此项目采用：
- 所有转换逻辑完全客户端执行
- 书籍工作台通过 `?bookId=` 切换
- Next.js 使用静态导出模式生成 `out/`

这样可以保持原有本地数据模型不变，同时直接部署到 GitHub Pages。

不走高饱和、重渐变、重阴影路线，保持简洁和内容可读性。

### 7. 开发态与构建态产物隔离

为避免 `next dev` 与 `next build` 互相污染，项目约定：

- `next dev` 使用 `.next`
- `npm run build` / `npm start` 使用 `.next-build`

这样可以避免：

- `routes-manifest.json` 缺失
- 开发页 500
- 热更新状态错乱
- 开发态代理 iframe 被构建产物干扰

## 风险与权衡

- IndexedDB 适合单机场景，但不具备跨设备同步能力
- 当前章节结构为单层列表，后续若支持分卷需要迁移数据模型
- `rehypeRaw` 预览策略适合本地工具，但若未来接入外部内容分享，需要重新评估安全边界
- Ant Design 主要承担交互弹窗职责，若后续扩大使用范围，需要进一步统一组件风格
- 正文提取采用启发式策略，不保证对所有站点稳定命中
- 强风控站点仍然可能无法自动采集，只能明确提示失败

## 抽象经验

以下经验与当前业务主题无关，可迁移到其他“外部内容导入”或“复杂预览导入”类功能中：

### 1. 先预览，再提交

只要导入动作会覆盖用户现有内容，就不应在抓取成功后直接提交。

更稳的模式是：

1. 先获取外部结果
2. 统一进入预览态
3. 由用户明确确认后再写入正式状态

### 2. 原始结果与预览状态分离

“抓取结果”不等于“用户最终要导入的结果”。

比较稳的做法是：

- 保留原始候选结果
- 再构造一个可编辑、可切换、可确认的预览状态
- 最终只让导入动作消费预览状态

### 3. 多路径输入，共用一个出口

如果系统同时支持多种导入方式，例如：

- 自动提取
- 手动框选
- 文件上传

那么最好让它们在后半段汇合到同一个预览和确认流程，避免每条路径都维护一套不同的提交逻辑。

### 4. 开发态与构建态隔离

任何依赖本地构建产物的框架，都应尽量避免让开发态和构建态共享同一个输出目录。

这是工程稳定性问题，不是业务逻辑问题。

### 5. 在开发环境中隔离嵌入式预览

当应用需要在 iframe 或类似容器中渲染外部 HTML 时，开发态脚本、overlay、HMR 往往会带来额外干扰。

比起直接导航，更稳的办法通常是：

- 把内容先抓成字符串
- 再以受控方式注入容器

### 6. 引导信息应分层

页面内默认文案只保留最短的行动提示；
更细的解释可以放到 Tooltip、帮助文档或二级说明里。

这样更利于保持界面干净，同时不牺牲可学性。
