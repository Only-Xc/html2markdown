# 通用布局与防溢出规则

**更新日期：** 2026-04-15

本文档沉淀与具体业务无关、但会高频影响工作台、抽屉、侧栏、预览面板体验的布局规则。

如果后续再做：

- 右侧抽屉
- 左侧侧栏
- 富文本预览
- 外部 HTML / Markdown 展示
- 图标按钮工具条

优先先对照本文件，避免重复出现“内容区太窄”“横向滚动条”“面板被子内容撑爆”的问题。

## 1. 内容优先的侧边面板宽度

当一个侧边面板同时承载表单、工具条、预览区、嵌入页面或富内容浏览时，它已经不是传统意义上的窄侧栏，而是“次级工作区”。

规则：

- 桌面端宽度优先按内容密度设置，而不是沿用组件默认窄宽度
- 移动端优先退化为 `100vw`
- 宽度优先使用 `min()`、`max()`、`clamp()` 之类的响应式表达
- 只有非常轻量的确认面板，才适合使用固定窄宽度

推荐：

- 内容型右侧抽屉：`min(880px, 100vw)` 到 `min(960px, 100vw)`
- 轻量操作型抽屉：`min(480px, 100vw)` 到 `min(640px, 100vw)`

## 2. 滚动边界必须分层

一个复杂面板里，不能让每一层都随意滚动。

规则：

- 最外层容器负责 `overflow: hidden`
- 真正承载内容的那一层负责 `overflow: auto`
- 不要同时让父层和子层都承担横向滚动
- 出现横向滚动时，先查是哪一层没有正确收口，而不是先怀疑内容本身

## 3. `min-w-0` 是工作区布局的默认检查项

在 `flex` / `grid` 布局里，只要某个子项既要伸缩、又要包住输入框、标签、工具条或富内容，就要默认检查 `min-w-0`。

常见信号：

- `flex-1`
- `minmax(0, 1fr)`
- 内部有 `Input`
- 内部有 `truncate`
- 内部有 HTML / Markdown 预览

缺少这个约束时，内容经常不会收缩，而是直接把父容器撑宽。

## 4. 富内容预览默认不可信

凡是渲染外部 HTML、Markdown、文章正文、代理页面片段，都不能假设内容天然“刚好适配容器”。

至少要补这些基础约束：

- 文本允许换行：`overflow-wrap: anywhere` 或 `word-break: break-word`
- 图片：`max-width: 100%`
- 视频 / iframe / embed / object：`max-width: 100%`
- 表格：允许自身内部横向滚动，不要撑外层
- `pre` / 代码块：允许自身内部横向滚动，必要时 `pre-wrap`
- 最外层预览容器：`min-width: 0`

## 5. 工具条优先可换行，其次才是压缩控件

顶部工具区在空间变窄时，优先顺序应是：

1. 允许换行
2. 允许标签截断
3. 控件宽度收缩
4. 最后才考虑隐藏非核心操作

不要一开始就把多个按钮硬塞在单行里，否则很容易把整个抽屉或卡片撑出横向滚动。

## 6. 富内容只允许在局部内部横向滚动

如果宽表格、长代码块确实需要横向滚动，应当发生在内容块自身内部，而不是：

- 整个抽屉横向滚动
- 整个页面横向滚动
- 父卡片被撑宽

判断标准：

- 用户滚动表格时，只滚表格
- 用户滚动代码块时，只滚代码块
- 面板本身宽度不因局部内容变化而跳动

## 7. 图标按钮必须补足可解释性

当工具条主要使用图标按钮时：

- 必须提供 Tooltip
- `button` 必须保持 `cursor: pointer`
- 禁用态如果仍需提示，外层需要可承载 Tooltip 的包装元素

这样可以在保持界面紧凑的同时，不牺牲可学性。

## 8. 变更时的检查顺序

后续再遇到“面板太窄”或“横向滚动条”问题，按这个顺序排查：

1. 面板宽度是不是按内容密度设置
2. 外层是不是已经 `overflow-hidden`
3. 中间层是不是缺 `min-w-0`
4. 顶部工具条是不是硬塞单行
5. 富内容区是不是缺最大宽度 / 换行 / 内部滚动规则

不要直接跳到“再加一点宽度”这种单点修补。
